Detailed Description

This part of the tutorial explaines how to create algorithm windows. Note that the above list contains many widget types that abstract from existing GUI toolkits. It allows algorithms to write high-level code that is independent of the GUI toolkit used, but only a core of typical GUI toolkits' features are available.

#include <fantom/algorithm.hpp>
#include <fantom/gui.hpp>
#include <fantom/register.hpp>
#include <mutex>
using namespace fantom;
namespace
{
class WindowTutorialAlgorithm : public VisAlgorithm
{
class MyWindow : public DockWindow
{
public:
MyWindow( MainWindow& mainWindow, const std::string& name )
: DockWindow( mainWindow, DockWindow::FFREE, name )
, mLayout( *this, false )
, mText( mLayout )
, mButton( mLayout, "Send", std::bind( &MyWindow::send, this ) )
{
}
void send()
{
if( mSendCallback )
{
mSendCallback( mText.get() );
}
}
void setSendCallback( std::function< void( const std::string& ) > callback )
{
mSendCallback = callback;
}
private:
BoxLayout mLayout;
LineEdit mText;
PushButton mButton;
std::function< void( const std::string& ) > mSendCallback;
};
void print( const std::string& name )
{
std::unique_lock< std::mutex > lock( mMutex );
infoLog() << name << std::endl;
}
std::mutex mMutex;
public:
class VisOutputs : public VisAlgorithm::VisOutputs
{
public:
VisOutputs( Control& control )
: VisAlgorithm::VisOutputs( control )
{
addWindow< MyWindow >( "Algorithm Window" );
}
};
WindowTutorialAlgorithm( InitData& init )
: VisAlgorithm( init )
{
getWindow< MyWindow >( "Algorithm Window" )
.setSendCallback( std::bind( &WindowTutorialAlgorithm::print, this, std::placeholders::_1 ) );
}
~WindowTutorialAlgorithm()
{
getWindow< MyWindow >( "Algorithm Window" ).setSendCallback( nullptr );
}
virtual void execute( const Algorithm::Options&, const volatile bool& )
{
}
};
AlgorithmRegister< WindowTutorialAlgorithm > dummy( "Tutorial/Window", "Demonstrate algorithm windows." );
}

When run, the algorithm shows a little window containing a line edit and a button. The window may be moved around and also docked inside the main FAnToM window. When the button is pressed, the current line edit's contents are shown in a message window.

#include <fantom/gui.hpp>

When using algorithm windows, the file "gui.hpp" needs to be included in addition to "algorithm.hpp".

class WindowTutorialAlgorithm : public VisAlgorithm

In order to create algorithm windows, you have to derive from VisAlgorithm.

class MyWindow : public DockWindow
{
MyWindow( MainWindow& mainWindow, const std::string& name )

This declares the nested class MyWindow. Algorithm windows must be derived from either DockWindow or MainWindow. We use DockWindow here. The window class does not have to be nested inside the algorithm that uses it, in this example it is only done to allow access to a private member of the algorithm. Note that the algorithm window always lives in the GUI thread whereas the algorithm itself lives in its own thread. Thus, any communication between algorithm and window needs to be synchronized!

: DockWindow( mainWindow, DockWindow::FFREE, name )
, mLayout( *this, false )
, mText( mLayout )
, mButton( mLayout, "Send", std::bind( &MyWindow::send, this ) )

This intializes the DockWindow and the window's members, i.e. its widgets. Refer to the list on the top of this page for details. All widgets require a reference to their parent widget as first parameter. Windows (MainWindow, DockWindow) and some special widgets (ScrollWidget) support only one child widget, but there are some widgets (BoxLayout, TabWidget, FormLayout) that are designed for multiple children and provide layout methods to arrange them in certain ways.

In this example, we want to display multiple widgets within a dock window, so we first add such a BoxLayout to it. This layout arranges its children in a row or a column next to each other. Then, the line edit and a button are added to the layout.

All GUI classes listed above support automatic memory management. You could simply add new widgets to a layout and forget about it (i.e., store neither pointer nor reference to it); when destructing the layout, all children are destroyed with it. This implies, that you should not attempt to destroy those children yourself afterwards. However, it is possible and recommended to construct the widgets in place within an enclosing class, similar to MyWindow, or use std::unique_ptrS for storing them and do memory handling explicitely. Then, the instantiation order as well as the destruction order of the widgets is crucial, as parents must be created before and destroyed after their children. If you do not use raw pointers, C++ will automatically destroy the widgets in the correct order (inverse to the construction order).

Many widgets have additional parameters like name, icon, and callback. The callbacks are executed when the user interacts with the widget, e.g. when the user clicks on a button, selects an item in a combo box, or changes the contents of a line edit. In the example, pressing a button will call the MyWindow::send() member function.

void send()
{
if( mSendCallback )
{
mSendCallback( mText.get() );
}
}

Algorithm windows have no way of knowing the algorithm they belong to. Therefore, the window can only send the data into the blue via a callback function. It is the responsibility of the algorithm to register a valid callback function and deregister it on destruction.

void print( const std::string& name )
{
std::unique_lock< std::mutex > lock( mMutex );
infoLog() << name << std::endl;
}

This function will be registered as the send callback later. When called from the algorithm window, it will run within the GUI thread. However, the infoLog() is also accessed from within the algorithm thread, e.g., within the execute() method. Therefore, a synchronization is necessary to prevent race conditions.

VisOutputs( Control& control )
{
addWindow< MyWindow >( "Algorithm Window" );
}

The algorithm window has to be declared in the VisOutputs of the algorithm. When the algorithm is instantiated, the window is instantiated as well.

WindowTutorialAlgorithm( InitData& init )
{
getWindow< MyWindow >( "Algorithm Window" )
.setSendCallback( std::bind( &WindowTutorialAlgorithm::print, this, std::placeholders::_1 ) );
}

When the algorithm is created, all visual outputs of the algorithm have already been instantiated. The window can easily be accessed via the getWindow() function. In this manner, the algorithm can register the print() function as the callback for sending data from the algorithm window to the algorithm. Note, that the algorithm is created within the algorithm thread. So, even within the algorithm constructor, you have to take care of sufficient synchronization when accessing or setting algorithm window members. For the callback no synchronization is needed, because the algorithm window only reads this variable and no race conditions can occur.
